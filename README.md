# Ludoxel v3.5

Ludoxel is a PyQt6 desktop application for controlled experimentation on a restricted voxel-world model, a first-person and third-person camera pipeline, and an OpenGL 4.3 Core Profile renderer. The present codebase contains a persistent flat sandbox, a separate Othello play space, state-dependent block-shape logic for a limited set of structural block families, gravity-affected falling blocks for `sand`, `red_sand`, and `gravel`, and a narrowly selected native-acceleration path for arithmetic-intensive kernels. The renderer now retains chunk-face instance storage incrementally and updates only the modified face ranges during world edits rather than rebuilding and re-uploading a merged world batch on every local interaction. That performance path presently depends on the generated in-place extension modules for `ray_aabb`, `voxel_dda`, and `view_angles`; if those binaries are absent, the application falls back to the Python sources for those kernels and the frame rate drops accordingly. It does not presently constitute a rule-complete reproduction of Minecraft Bedrock Edition or Minecraft Java Edition, and it should not be represented as a finished system.

The repository should instead be read as an engineering workbench. The implemented systems are those that are currently useful for rendering, collision, picking, input, persistence, and deterministic numerical inspection. Where Minecraft-derived semantics are present, they are local design targets for specific subsystems rather than a claim of full game equivalence across movement, combat, world generation, inventory logic, redstone, networking, or content coverage.


## Etymology

Latin **ludus** underlies the initial element **lud-**. Its attested semantic field extends across play, game, sport, and school or training. The stem therefore bears a general ludic reference: not combat in particular, not one ruleset in particular, and not training in isolation, but play taken in its broader and more exact genus.

**Voxel** is the modern technical contraction of **volumetric** and **pixel**. In technical usage, it denotes a discrete element of three-dimensional representation, commonly treated as the spatial analogue of the pixel, and thus refers to discretized volume rather than merely to visual style or atmospheric motif.

**Ludoxel** accordingly denotes ludic activity in voxel space, or, more strictly, play conducted within a discretized volumetric world. As the title of a sandbox application, the term is exact in the only sense that matters here: the operative environment is voxel-constituted, while the activity admitted within it is ludic in a broad sense, namely as play not exhausted by any single closed game form, but proceeding through locally open course and user-directed manipulation.


## Application modes

Ludoxel currently exposes two persistent application modes inside a single desktop shell: `My World` and `Play Othello (Reversi)`.

### Switching between spaces

The pause menu exposes two persistent destinations. `Play My World` returns to the sandbox space and is disabled while that space is already active. `Play Othello (Reversi)` transfers the session into the dedicated Othello space and is disabled while the user is already there. Each space persists its own player transform and session-specific state. The pause overlay is organized as a screen-wide two-column surface. The left half contains the boxed session and space-management controls. The right half contains a vertically centered, unboxed live slim-arm player preview with the skin import and reset actions placed immediately below it. The preview interaction path only supplies the desired view angles. The actual body image is rendered through the same OpenGL player-model pass and skin texture path used by the in-world renderer and is then composited onto the pause surface without introducing a second UV or cube-layout implementation. The preview framing is biased slightly downward and farther back so that the full head silhouette remains inside the visible image area. Pointer motion across the entire pause surface, including both columns, steers the head orientation whenever the left mouse button is not held. Holding the left mouse button and dragging anywhere on the pause surface rotates the preview body around the vertical axis. The head yaw remains a bounded offset from the current body heading, and the pitch follows the screen-space vertical direction rather than inverting it.

### `My World`

`My World` is the ordinary persistent sandbox space. It serves as a flat inspection environment in which the implemented block subset can be placed, broken, rendered, picked, and collided against. The current catalog includes full cubes together with slabs, stairs, fences, fence gates, and walls, each driven by explicit block-state logic for render boxes, collision volumes, pick volumes, and structural connectivity. `sand`, `red_sand`, and `gravel` are gravity-affected blocks and now leave the static world as transient falling bodies whose descent is advanced on a Minecraft-like falling-block tick and rendered continuously between ticks rather than being teleported cell-by-cell through the chunk mesh. Their landing path now also treats lower stairs, fences, fence gates, and walls as valid resting supports for the full falling block, while a collision with a lower slab destroys the falling block and emits the same block-break particle effect that ordinary manual block breaking uses. Block destruction now also emits short-lived terrain-fragment break particles whose spawn positions are sampled from the broken block's active render boxes rather than from a hard-coded full-cube assumption. Those particles are now emitted through stochastic volume sampling rather than a uniform lattice, so thin members and partial-volume blocks no longer produce visually uniform fragment rows. The player collision path now includes depenetration for pre-existing overlaps so that block edits do not leave the camera embedded in geometry, while closed fence gates preserve the Minecraft-like special case in which a player already standing inside the gate is not forcibly expelled when the gate is shut around them, and landed gravity blocks likewise preserve the overlap when they settle onto the player rather than pushing the player aside. Persisted-state restoration now reconstructs those overlap exemptions before the first post-load collision step, which eliminates the previous restart case in which a player saved inside a closed fence gate was lifted onto the gate on relaunch. The world model remains intentionally narrow when measured against the breadth of a complete Minecraft implementation.

Within this space, the ordinary block interaction path remains active. Holding the left mouse button in creative mode now repeats block breaking continuously, and holding the right mouse button now repeats both block placement and fence-gate interaction continuously. The persistent default cadences are `0.30 s` for break repetition, approximately `0.008 s` for placement continuation, and `0.20 s` for interaction repetition. Those interaction cadences are stored with the other runtime preferences and can be edited from the `Game Player` settings page. Continuous right-click placement is now constrained to a single aligned continuation line while the button remains held, and the maintained line can grow in either signed direction from the first placed block instead of being hard-locked to one side. The held-placement repeater now preserves the initial right-click placement as a single isolated action and only arms after an initial `0.20 s` dwell, after which the held `place` path resumes on the existing high-cadence continuation path before the player collision step of each simulation tick. This removes the remaining accidental two-block burst on ordinary single clicks without sacrificing the sustained bridging cadence or forcing jump-bridge support to wait until after gravity resolution. When the player begins a held sequence from a directly visible horizontal side face, that sequence enters a dedicated face-chain mode whose continuation axis remains locked to the same face orientation of the frontier block. When the player instead begins from supporting ground without such a qualifying visible frontier face, the placement path synthesizes the same side-face basis that would result from placing against that supporting block in the current facing direction, and grounded crouch bridging now prefers that horizontal reacharound basis over accidental top-face or bottom-face hits that would otherwise redirect the action into upward or downward stacking. That start-of-sequence support-face classifier now accepts any hit on the supporting block that lies on the intended side half, rather than only a narrow outer-edge strip, so an aim that is already committed to that side no longer needs an unusually exact literal side-face pick to enter the side-route continuation. When that supporting-block hit still resolves to the top or bottom face, the classifier now reconstructs the intended horizontal side from the actual top-surface hit displacement whenever one horizontal axis dominates the local corner geometry, which prevents a clearly side-committed corner aim from falling back to the raw facing direction and emitting an unwanted forward wall before the side route exists. A vertical continuation is now admitted only while that held sequence is still at its first follow-up decision and only while the player remains horizontally co-located with the original two-block anchor pair formed by the first placed block and its immediately preceding support block, with live support or pick context on that same pair also still counting as anchored evidence. For horizontal-face starts, that preceding support block is still the immediately previous block along the maintained horizontal continuation axis. For top-face and bottom-face starts, it is now the actual block that was clicked to produce the first upward or downward placement, and the vertical branch is accepted only when the commanded vertical sign matches that starting face orientation. When the initial right-click cannot place because the target cell is still empty in the world but is temporarily occupied by the player body, the repeater now still arms from that clicked face and treats the unresolved `hit.place` cell as the starting cell of the held route. That deferred route now retries that same unresolved start cell against the originally clicked support face until the block genuinely materializes there, and a successful materialization immediately preserves the matching upward or downward continuation state instead of discarding that vertical command until a later retry. The maintained frontier now advances only when the attempted target cell actually remains a valid world block at that exact cell after the placement decision, so a failure caused by player overlap, missing support, or any other rejected state leaves the repeat progress unchanged instead of skipping forward as though the absent block already existed. The same frontier gate is applied to gravity-affected placements as well, which prevents a newly placed unsupported falling block from being treated as the next support in the held route before it leaves the static world and terminates that held continuation after the single emitted falling block instead of re-emitting the same unsupported placement indefinitely. That restriction prevents a bridge or side-face chain that has already been established from mutating later into a tower rooted at an obsolete start cell, while still allowing the first follow-up placement to resolve upward or downward without requiring a separate forward wall. The visible-face, support-face, and generic horizontal repeat routes now consult the live movement command projected onto the maintained horizontal continuation axis instead of a bare nonzero horizontal-input test. They therefore refuse to advance the forward frontier whenever the held sequence is not actually commanding motion along that axis, and under that condition the first follow-up decision remains available to the upward or downward branch. Once that vertical branch has been recognized, the repeater now persists the reoriented vertical line even across a no-placement retry, so the next tick no longer falls back to the previous horizontal route merely because the player had not yet crossed the next placement threshold on the first branch-detection tick. The first detected world-space displacement away from the hold origin now locks the entire held vertical branch to that sign, so a hold that has once committed upward no longer emits downward placements and a hold that has once committed downward no longer emits upward placements. Once such a vertical line has been committed, further continuation advances only toward the next neighboring frontier cell in that locked direction from the player's signed feet-height displacement measured against the hold origin rather than from the changing view ray, and if the player outpaces that immediate neighboring frontier cell by more than one block the held path no longer back-fills the skipped vertical cells. That support-face continuation otherwise no longer free-runs while the view remains locked to the original supporting block face. It now evaluates both the directly visible frontier face and a forward projected frontier-face intersection against the maintained bridge direction, but that projected frontier path remains unavailable while the player is still grounded on the pre-advance support block and is only admitted once the player has either gone airborne or has already transferred support onto the present frontier block, with the additional requirement that the current pick ray has already advanced onto that frontier block rather than still resolving to an older support block. When the live view becomes nearly vertical, the effective interaction pitch remains clamped inside the same finite domain used by the player look state so that near-vertical camera shake cannot flip the rendered image or invert the horizontal intent that the support-face classifier derives from the current view. When a qualifying frontier face is directly visible during that support-face sequence, that visible face still takes precedence over the synthesized support candidate and the continuation proceeds only through that same face orientation, except when the visible face still belongs to the block presently supporting the player or when the visible face reverses the maintained bridge-face orientation. For ordinary visible side-face continuation that is not in this face-chain path, and for generic horizontal continuation started from non-side-face placement, the first follow-up vertical branch now also accepts an upward start from the live jump command itself and no longer waits exclusively for the already-integrated displacement sample. The same first-follow-up vertical branch is now available from top-face and bottom-face starts as well, instead of being limited to side-face starts. Wall top-support updates now also account for the presently implemented block families above the wall. A straight wall run that would otherwise remain flat now reintroduces its center post when the directly overlying block is another wall or a fence, while a directly overlying full cube, fence gate, slab, or stairs block instead promotes the supported wall arms to full-height tall segments and preserves the flat center when opposite tall supports are formed, which matches the implemented Minecraft-style wall top-support distinction for the available block subset. The block inventory is opened with `E`, world edits persist with the sandbox space, and re-entering the space restores the previous player transform together with local world modifications.

Camera perspective is part of the persistent runtime state. The default cycle order follows the Minecraft-style `F5` sequence `First Person -> Third Person Back -> Third Person Front -> First Person`, the same action is remappable through the keybind settings, and the current perspective can also be selected explicitly from the video settings page. Third-person camera placement is collision-constrained against block collision volumes, so the camera retracts before penetrating nearby geometry and does not expose interior block faces by clipping through them. The third-person world projection now also uses a smaller near plane together with a larger camera-clearance margin, which keeps nearby block faces visible instead of allowing them to vanish at the lens boundary while the camera is pressed back against geometry. The gameplay crosshair is suppressed outside first person. The third-person player body and the first-person arm now use a persistent slim-arm skin texture. The default texture remains Alex, a custom modern `64x64` skin can be imported from the pause menu, the imported file is overwritten in place under `configs`, and resetting returns the renderer to Alex. Both held blocks and held Othello control items follow the same swing state in the visible model and in the shadow caster. Step transitions now drive the rendered player base height with the same vertical easing used by the camera eye, which removes the previous shadow-only teleport impression while ascending or descending shallow ledges.

Video settings are now hosted in a separate application-modal settings window rather than inside an in-viewport overlay, so visual changes can be inspected against the live world while the window remains open. That window still exposes the default Minecraft-style crosshair together with a persistent `16x16` pixel editor. A custom pattern can be drawn with the left mouse button, erased with the right mouse button, and reset through `Clear Board`, which simultaneously clears the editor board and restores the default Minecraft-style crosshair. The gameplay crosshair is rendered without the previous forced black outline, regardless of whether the active pattern is the default form or a user-authored bitmap. Cloud density and cloud seed now follow the same enabled-state semantics already used by view bobbing and camera shake, so disabling clouds visibly disables those sliders rather than leaving stale editable values on screen. The `Video` page now also exposes persistent break-particle spawn-rate and break-particle speed controls, while the `Game Player` page retains the persistent block-break and block-place repeat interval controls alongside the movement parameter controls. In `My World`, the creative inventory now also includes a live search field for block-name and block-id filtering, so large catalog scans no longer require manual visual paging.

When the detached settings window or the detached Othello settings window is opened while fullscreen is enabled, the host window now returns temporarily to the normal state for the duration of that detached dialog and reapplies the stored fullscreen preference only after the detached window has been closed. This preserves access to the dialog controls without forcing the user to lose the persisted fullscreen preference merely to edit settings.

### `Play Othello (Reversi)`

`Play Othello (Reversi)` is a second persistent play space hosted inside the same application shell. It has its own hotbar, match settings, AI opponent, clocks, board interaction path, piece animation path, and rendering path. This mode is a specialized subsystem layered onto the renderer and UI framework. It should be understood as a self-contained application mode rather than as evidence that the surrounding sandbox has reached general game completeness.

Within this space, block placement, block breaking, and the block inventory overlay are disabled. The Othello hotbar is reserved for control items. Slot `1` contains `Start`, and slot `9` contains `Settings`. Selecting either slot now equips a dedicated enlarged control icon rather than leaving the hand empty. The icon is rendered as a single visible face in first person and is also attached to the third-person player model and its shadow so that the active control item remains legible before use. Right-clicking `Start` begins a fresh match or restarts the current match under the stored defaults. Right-clicking `Settings` opens a separate application-modal Othello settings window, which configures AI strength, time control, disc-animation mode, player order, sacrifice level, worker count, hash level, and opening-book learning limits for the next match. The difficulty menu exposes `Weak`, `Medium`, `Strong`, `Insane`, and `Insane+`. `Insane` remains the cached bitboard-search engine without enforced opening-book selection, while `Insane+` is the higher tier that consults the bundled and user-extended opening book before falling through to the same search core. The time-control menu now exposes `Timer off`, `1 move 5 seconds`, `1 move 10 seconds`, `1 move 30 seconds`, `1 minute per side`, `3 minutes per side`, `5 minutes per side`, `10 minutes per side`, and `20 minutes per side`. The animation menu now exposes the previous simultaneous-flip path as `Animation off` together with `Ripple fast` and `Ripple slow`, both of which start successive flips from the placed disc outward instead of rotating every captured disc at once. The settings window can extend the opening book by launching threshold-controlled learning with explicit depth, per-move error, cumulative error, and leaf-error bounds. That learning run now exposes a cancellable progress state, mirrors its current status inside the detached settings window and the in-world Othello title HUD, temporarily projects the currently explored learning position onto the rendered board, and preserves partial progress on cancellation because user-generated book lines are written persistently under `configs/othello_opening_book.json`. The same settings window also supports opening-book import and export. Book storage is symmetry-aware at the position level through canonicalized board transforms, so search lookup and learning-state pruning treat rotated and reflected equivalents as the same opening-book position. The default Othello configuration remains `Medium`, `20 minutes per side`, `Animation off`, and `player moves first`.

Disc placement is performed by aiming at the board and pressing the left mouse button on a legal square. Hovering now reports the algebraic square name, such as `a1` through `h8`, and the Othello HUD has been reworked into the same panel style used by the normal `F3` debug HUD. That Othello HUD is hidden whenever the ordinary `F3` HUD is shown, and otherwise reports the current turn, best move, principal line, player-versus-AI evaluation, and a textual depth graph derived from the current search trace. The rendered board is hosted in a separate persistent world space, and the match clocks pause while the pause menu or the separate Othello settings window is open.


## Source layout and execution model

The repository intentionally preserves the existing `python -m main` startup route. `main.py` prepends `src` to `sys.path`, after which control passes into `ludoxel.application.boot`. The public project title is **Ludoxel**, whereas the current source package and import namespace remain `ludoxel`. This separation is intentional at the present stage and allows the visible application identity to evolve without forcing an immediate package-level rename across the codebase.

At startup, the shell now presents a dedicated top-level loading screen before the main window is allowed to surface, and the embedded viewport still retains its in-window loading overlay until renderer initialization has completed and the initially visible chunk set has reached GPU residency. Both loading surfaces now search the bundled UI assets for `assets/ui/ludoxel.png`, `assets/ui/ludoxel.svg`, `assets/ui/ludoxel.jpg`, `assets/ui/ludoxel.jpeg`, `assets/ui/ludoxel.webp`, and `assets/ui/ludoxel.bmp` in that order and render the first decodable file as the title mark, while falling back to the textual `Ludoxel` title when no such asset is present. Simulation input remains suspended during that phase, so the user does not observe the previous in-world hitching while the initial chunk uploads are still converging and the launch path no longer leaks an initial blank frame from the not-yet-initialized OpenGL surface. The persisted window rectangle and target monitor are restored before either the splash screen or the main window is shown, so relaunching returns the application to the previous display and window size rather than recentering it on the primary monitor. Space transfers between `My World` and `Play Othello (Reversi)` reuse a retained CPU-side chunk-build cache and immediately re-enter the same loading overlay while the target residency set is unresolved. When the desktop shell becomes inactive, simulation input is suspended, mouse capture is released, and the pause menu is opened explicitly even when the deactivation route is an operating-system escape such as the Windows key. Time-based renderer motion such as cloud drift and animated-texture stepping is paused for true application deactivation and for the pause, death, inventory, and Othello-settings states, while the separate modal settings window remains live so that visual settings can be inspected against the world in real time. When the shell becomes active again, the viewport explicitly re-arms the next world upload and selection refresh so that returning from PAUSE or INVENTORY does not wait on stale render cadence before the world resumes stable drawing. Mouse-look capture now also enters a short resynchronization phase after capture is re-enabled. During that phase the system cursor is repeatedly re-centered, keyboard movement continues to flow, camera delta is forced to zero, and normal mouse-look sampling is not restored until the cursor has remained at the viewport center across successive polls. The pause-screen player preview now preserves its last head orientation when the pointer leaves the application and resumes direct pointer tracking immediately when the pointer re-enters the application instead of running a delayed reacquisition phase.

The source tree is now organized by responsibility. `ludoxel.application` now contains only shell bootstrap, session/runtime orchestration, persistence, and audio; platform diagnostics moved out to `ludoxel.shared.diagnostics`, and the renderer-facing camera, player-model, and falling-block snapshot DTOs now live under `ludoxel.shared.rendering.render_snapshot` because they are shared rendering contracts rather than application state. `ludoxel.shared` contains the reusable math, world, block, systems, rendering, OpenGL, diagnostics, and Qt layers, and Othello-specific code remains isolated under `ludoxel.features.othello`. Within the shared UI layer, genuinely reusable status surfaces remain under `ludoxel.shared.ui.common`, viewport-local input, overlay, selection, and frame-synchronization helpers live under `ludoxel.shared.ui.viewport.runtime`, and settings-specific controls now live under `ludoxel.shared.ui.settings.widgets`, while the detached settings surface keeps page construction and widget-value synchronization inside `ludoxel.shared.ui.settings.settings_overlay` instead of spreading one private implementation across multiple helper files. Within the CPU-side render-preparation layer, face visibility tests, UV selection, row packing, falling-block row emission, and chunk face-payload assembly are grouped under `ludoxel.shared.rendering.faces` rather than being flattened across the rendering package root. Othello rendering state and scene builders now live under `ludoxel.features.othello.rendering`, search and worker code live under `ludoxel.features.othello.domain.engine`, opening-book persistence and learning live under `ludoxel.features.othello.domain.book`, immutable bundled book data lives under `ludoxel.features.othello.resources`, and the remaining `ludoxel.features.othello.domain.game` package is reserved for board geometry, rules, and game-state types. This keeps the common renderer, CPU-side render preparation, shared world model, and shared Qt surface separate from feature-local gameplay code. Reusable scalar clamping, interpolation, and persistence-safe scalar coercion helpers are consolidated under `ludoxel.shared.math`, while the textured-face rendering core remains centralized in `ludoxel.shared.opengl.passes.textured_face_pass` and the first-person, falling-block, held-block, and special-item entry points remain split into dedicated pass modules so that each hot-path wrapper keeps one narrowly scoped responsibility; UI SVG assets that are consumed outside the package tree live under the repository-level `assets/ui`.

The native build remains deliberately narrow. Only `ludoxel.shared.math.geometry.ray_aabb`, `ludoxel.shared.math.voxel.voxel_dda`, and `ludoxel.shared.math.view_angles` are compiled in place. Those modules are dominated by scalar arithmetic, geometric branching, and dense numerical work. Broader scene and block orchestration layers remain in Python because they are governed primarily by Python object traffic, callback dispatch, dictionary access, and heterogeneous container traversal, all of which reduce the benefit of opaque extension-module compilation and can worsen end-to-end frame cost when moved wholesale across the extension boundary. The supported recovery path for that acceleration layer is now the repository script `scripts/build_native_extensions.py`, which rebuilds the three generated extension modules in place and verifies that the interpreter resolves those imports to compiled binaries instead of the Python fallback sources.

The editable install and the explicit native build are intentionally separated. The editable install makes the source tree importable for development. The explicit in-place native build performs the optional native compilation step through `scripts/build_native_extensions.py`, which delegates to `setup.py build_ext --inplace` and then verifies the imported module targets. This separation prevents editable-install metadata evaluation from depending on Cython at the moment when development dependencies may not yet be installed.

For the ordinary Windows executable path, `scripts/build_windows_exe.py` is the primary entry point and `scripts/build_native_extensions.py` is usually not invoked manually because the executable builder already runs that narrower native rebuild unless `--skip-native-build` is supplied. The standalone `scripts/build_native_extensions.py` path remains intentional for source-tree development, for performance recovery after `scripts/reset_build_artifacts.py`, and for any case in which the in-place `.pyd` modules are needed without rebuilding the Windows bundle.


## Development workflow

The development path described below assumes Python 3.14.x, PyQt6 6.6 or newer, NumPy 1.26 or newer, PyOpenGL 3.1.7 or newer, and a working OpenGL 4.3 Core Profile context. On Windows, the Microsoft Visual Studio Build Tools C++ workload is assumed to already be installed before any local native build is attempted. The repository-local bootstrap now also prefers a sibling `Python314` interpreter when `main.py` or the package entry point is invoked from a different local CPython on Windows, because the optional in-place native hot-path extensions are version-specific and are expected to match the stated 3.14 runtime.

The following sequence is the intended Windows development path for a clean environment.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip setuptools wheel
python -m pip install -e ".[dev]" --no-build-isolation
python .\scripts\build_native_extensions.py
python -m ludoxel
```

After the in-place native build has completed on Windows, the verification step inside `scripts/build_native_extensions.py` should resolve all three hot-path imports to `.pyd` files rather than the corresponding `.py` sources. If the native build is not required for the task at hand, the application can still be launched through `python -m main` after the editable install, but the renderer will then use the slower Python fallbacks for those kernels.

If one of the native-accelerated modules is edited, rerun the in-place extension build.

```powershell
python .\scripts\build_native_extensions.py
```

If packaging metadata, dependency declarations, entry-point definitions, or package-data rules are changed, rerun the editable install as well.

```powershell
python -m pip install -e ".[dev]" --no-build-isolation
```

On Windows, a loaded `.pyd` file cannot be overwritten while a Python process still holds the module image open. If the in-place native build fails with a file-in-use or permission error, terminate the running application and any interactive Python processes that imported the target module, then rerun the build.

```powershell
taskkill /F /IM python.exe
python .\scripts\build_native_extensions.py
```

The repository-local cleanup script now removes those generated `.pyd` files deliberately. After `scripts/reset_build_artifacts.py` has removed them, rerun `scripts/build_native_extensions.py` before the next performance-sensitive session, because until that rebuild has completed the application will again import the slower Python fallback kernels.

```powershell
python .\scripts\reset_build_artifacts.py
python .\scripts\build_native_extensions.py
```

Standard source and wheel packaging remains available through the ordinary frontend.

```powershell
python -m build
```

The supported Windows executable packaging path is now the repository script `scripts/build_windows_exe.py`. That script first rebuilds the native hot-path extensions unless `--skip-native-build` is given, then invokes PyInstaller in one-file mode, collects the package data that lives under `src/ludoxel`, and bundles both the repository-level `assets/` tree and the repository-level `src/` tree into the PyInstaller data root that the frozen process resolves through `sys._MEIPASS`. The application therefore loads immutable bundled resources from that internal extraction root while keeping runtime writes confined to `configs/` beside `Ludoxel.exe`, and no first-launch synthesis of `assets/` or `src/` beside the executable is part of the supported path. PyInstaller onefile still performs its own temporary extraction into the operating-system temporary directory before control reaches the application, so the deployment surface exposed next to the executable remains a single `Ludoxel.exe` together with the user-writable `configs/` tree. The builder now also removes the legacy `dist\windows\Ludoxel` onedir output on a best-effort basis before and after the onefile build so that an obsolete `_internal` tree from an older packaging mode does not remain visible beside the new single-file output. The builder now allocates run-unique PyInstaller work and spec directories under `build/pyinstaller-runs` and `build/pyinstaller-spec-runs`, and it also allocates a run-unique staging distribution directory under `build/pyinstaller-dist-runs`. The PyInstaller build always targets that staging directory first, after which the script attempts to publish the staged `Ludoxel.exe` into `dist\windows\Ludoxel.exe`. If the published path is locked because the previous executable is still running, the script now preserves the staged executable instead of failing and reports that staged path explicitly.

If a custom application icon is desired, place it under `assets/ui`. The canonical Windows executable icon path is `assets/ui/app_icon.ico`. When that file exists, `scripts/build_windows_exe.py` forwards it to PyInstaller through `--icon`, so the generated `Ludoxel.exe` carries the custom shell icon. The running Qt application also searches the bundled resource root for `assets/ui/app_icon.ico`, `assets/ui/app_icon.png`, `assets/ui/app_icon.jpg`, and `assets/ui/app_icon.jpeg` in that order and applies the first decodable file as the runtime window icon, but only the `.ico` file is suitable for embedding into the Windows executable itself.

The frozen Windows entry path now also enables multiprocessing freeze support before control reaches the application shell, assigns an explicit Windows AppUserModelID for the Ludoxel desktop identity, and suppresses visible console windows for the diagnostic `tasklist` and `nvidia-smi` probes that may be used by the optional debug HUD. This keeps transient internal worker or probe activity from surfacing as visible terminal flashes or from degrading icon resolution on the taskbar and related shell surfaces.

```powershell
python .\scripts\build_windows_exe.py
```

The default output path is `dist\windows\Ludoxel.exe`.

That packaging path remains intentionally distinct from the explicit local optimization path. The repository does not make Cython execution an implicit side effect of every packaging or installation action, but the supported Windows bundle builder now executes the narrow native build automatically because the measured renderer frame rate depends materially on those three compiled kernels being present.
